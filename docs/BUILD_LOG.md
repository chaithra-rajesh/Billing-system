## 2026-05-09 ‚Äî auth: 30-day session inactivity timeout

**What changed:** Added `[auth.sessions]` block to `backend/supabase/config.toml` with `inactivity_timeout = "720h"` (30 days). Refresh tokens become invalid if the user hasn't used the app within 30 days. `timebox` is intentionally unset ‚Äî active users stay logged in indefinitely (no hard cap).

**Why:** Default refresh tokens never expire, which is too permissive for a system that touches money + GST data. 30 days balances security against user friction (re-login every 30 days of inactivity is acceptable).

**Files:** `backend/supabase/config.toml`.

**Follow-ups:** This setting only takes effect locally. **Production must be set in the Supabase dashboard** at Auth ‚Üí Sessions ‚Üí Inactivity timeout = 30 days. (Note: this is a Pro-plan feature on managed Supabase; on Free, refresh tokens never expire.) Local Supabase needs `supabase stop && supabase start` for the change to apply.

## 2026-05-06 ‚Äî User management: invite, role, per-franchise active toggle

**What changed:** Super admins and franchise admins can now manage users from the franchise Settings page. New "Users & permissions" section lets them invite a new email (sends a Supabase Auth invite; the auth-sync trigger seeds `public.users`), add an existing platform user to this franchise, change a user's role between `billing_user` and `franchise_admin`, and revoke their membership without affecting the user's other franchises or their global account. Inviting an email already on the system just adds them to this franchise ‚Äî that's also how an admin "adds another franchise to an existing user".

**Backend:**
- Migration `20260506000004_user_management_rls.sql` ‚Äî fixes the broken `ufr_select` policy (`user_id = auth.uid()` stopped working when public.users.id was decoupled from auth.users.id in 0505000002 ‚Äî replaced with the proper `auth_user_id` join), and re-introduces the franchise_admin branch on `ufr_insert`/`update`/`delete` using the recursion-free `is_franchise_admin()` helper from migration 0506000002.
- New edge functions:
  - `invite-user` ‚Äî find-or-invite by email; on miss calls `supabase.auth.admin.inviteUserByEmail()` and waits for the auth-sync trigger to create `public.users`; then upserts a `user_franchise_roles` row with the requested role.
  - `list-franchise-users` ‚Äî joins `user_franchise_roles ‚Üí users` (service client, since users RLS hides peers from non-admins) and returns one row per membership: `{ ufr_id, role, ufr_active, granted_at, id, email, full_name, user_active }`.
  - `update-user-membership` ‚Äî partial PATCH on a single ufr row (role and/or is_active); user-scoped client so the new RLS policies enforce authorization.

**Frontend:**
- New `frontend/src/features/users/{api.ts, hooks.ts, components/users-section.tsx}` ‚Äî typed API + TanStack Query mutations + a self-contained section component.
- Settings page renders `<UsersSection franchiseId={...} />` below the Logos section.

**Why:** Until now, granting franchise access required a SQL `insert` against `user_franchise_roles`. That doesn't scale past the seeded super_admin. The global `users.is_active` flag stays super-admin-only by design ‚Äî disabling it blocks login across every franchise the user belongs to, which is a power a single franchise admin shouldn't hold. Per-franchise revocation via `ufr.is_active = false` covers the realistic admin use case (remove from this franchise) without that side-effect.

**Files:**

## 2026-05-06 ‚Äî Standard pagination across every list-* edge function

**What changed:** `list-invoices`, `list-customers`, and `list-my-franchises` now share one pagination contract: each accepts `limit?` (1‚Äì200, function-specific defaults) and `offset?` (‚â•0), and returns `{ items..., total, limit, offset }`. The invoices list page consumes it with Prev/Next + "X‚ÄìY of N" ‚Äî pagination state lives in component state and `placeholderData` keeps the previous page visible while the next one streams in.

**Why:** The lists were a mix: invoices already paged but didn't echo limit/offset; customers had a `limit` cap (5) but no offset/total; franchises returned the full set every time. Different shapes meant the frontend had to special-case each. Aligning the contract lets the same `<Pagination>` UX cover all three and keeps payloads bounded as data grows.

**Design notes:**
- Server caps `limit` (200 across the board) so a malicious or confused client can't request the world. Defaults are tuned per resource: invoices=25 (table view), customers=50 (admin), franchises=50 (picker).
- For invoices and customers, `total` comes from PostgREST's `count: 'exact'` on the same query ‚Äî one round-trip, accurate after filters.
- For non-super franchises, the membership join can't combine `is_active` filter on the joined franchise with an exact count, so we fetch all memberships for the user and slice in JS. Practical hit: even users with 100 franchise roles are well under 100KB.
- The bold field in CLAUDE.md rule 9 (idempotency) doesn't apply to GET-shaped reads, so no idempotency-key plumbing here.

**Files:**
- `backend/supabase/functions/list-invoices/index.ts`
- `backend/supabase/functions/list-customers/index.ts`
- `backend/supabase/functions/list-my-franchises/index.ts`

## 2026-05-06 ‚Äî PDF font registration

**What changed:** `invoice-pdf.tsx` registers Roboto (Apache 2.0) at module load via `Font.register`, served by jsdelivr's `@fontsource/roboto@5` CDN as WOFF. The same font is also registered under the alias families `Helvetica` and the literal CSS-stack `Helvetica, Arial, sans-serif`, so any default fontFamily that bypasses our explicit `'Roboto'` resolves to the same glyphs instead of throwing.

**Why:** `@react-pdf/renderer` v4 still ships Helvetica metrics in the browser bundle, so a plain `fontFamily: 'Helvetica'` *should* work ‚Äî but a Text node was somehow ending up with the literal CSS-stack value (`"Helvetica, Arial, sans-serif"`), which is strict-string-matched in `font/getFont()` and threw. Aliasing the same source under those names is the bulletproof fix and avoids playing whack-a-mole with cascade leaks.

**Files:**
- `frontend/src/features/invoices/components/invoice-pdf.tsx`

## 2026-05-06 ‚Äî Audit trigger user_id lookup fix + customer typeahead

**What changed:**
1. New migration `20260506000001_audit_user_id_lookup.sql` rewrites `audit_trigger_fn` to look up `public.users.id` via `auth_user_id = auth.uid()` instead of writing `auth.uid()` straight into `audit_logs.user_id`. Without this any authenticated INSERT/UPDATE/DELETE on an audited table failed with `audit_logs_user_id_fkey`.
2. The Bill-To Party picker on the new-invoice page is now a server-search typeahead capped at 5 results, not a full-list `<select>`. `list-customers` accepts a `limit` (default 50, max 50) and the editor debounces input by 200ms, only fetching matches as the user types.

**Why:**
- Audit FK regression: when `auth_user_id` was added to `public.users` in migration `‚Ä¶0505000002`, the two id spaces were split (public.users.id ‚âÝ auth.users.id from then on). The audit trigger predated that change and was never updated, so any row write by an authed user blew up against the `audit_logs.user_id` FK back to `public.users(id)`. Surfaced when the new "+ Add new customer" flow tried to insert a customer.
- Performance / UX: franchises will accumulate hundreds of customers; loading the entire list to populate a `<select>` is wasteful and the picker becomes unusable. Server-side filter + 5-row results keeps both payloads and the dropdown small.

**Design notes:**
- Trigger maps `auth.uid()` ‚Üí `public.users.id` and falls back to NULL for service-role writes (no app-user context). `audit_logs.user_id` already allows NULL (`on delete set null`), so unattributed system writes get logged with a null user instead of erroring.
- Typeahead holds the picked customer in component state (`selectedCustomer`) rather than re-deriving from the search results ‚Äî once a row is chosen it stays displayed even as the search text changes. The hidden `customer_id` input keeps RHF's `required` validation working.
- Add-new flow now pre-fills the draft name with whatever the user already typed in the search box, so they don't retype it.

**Files:**
- `backend/supabase/migrations/20260506000001_audit_user_id_lookup.sql` (new)
- `backend/supabase/functions/list-customers/index.ts` (added `limit` param)
- `backend/supabase/functions/create-customer/index.ts` (new ‚Äî earlier in this session)
- `frontend/src/features/customers/api.ts` (`listCustomers` options object; `createCustomer`)
- `frontend/src/features/customers/hooks.ts` (`useCustomers` options object; `useCreateCustomer`)
- `frontend/src/app/franchises/[slug]/invoices/new/page.tsx` (typeahead + add-new flow)

**Follow-ups:** Same inline-add pattern would help the Ship-To Party block. The list-customers `limit` cap of 50 may need to grow if/when an admin "all customers" page is built ‚Äî that consumer should use a paginated query instead of bumping the cap.

## 2026-04-29 ‚Äî Backend slice: read EFs + create-invoice + bootstrap

**What changed:** First Edge Functions on disk. Added a shared handler wrapper, an idempotency helper, an Indian-numbering words helper, four read endpoints (`list-my-franchises`, `list-customers`, `list-invoices`, `get-invoice`), and the core `create-invoice` mutation. **Why:** The product flow we agreed on (login ‚Üí franchise picker ‚Üí list ‚Üí new bill ‚Üí PDF) needs read EFs to populate every screen and `create-invoice` to actually persist. Building the contracts first means the UI work next turn is just data-binding.

**Design notes:**
- `_shared/handler.ts` swallows the boilerplate every EF shares (preflight, JWT auth, JSON parsing, AppError ‚Üí JSON). Each function is just the inner handler.
- `_shared/idempotency.ts` hashes (function name + canonical body) and stores the result keyed by (`user_id`, `Idempotency-Key`). Replays return the cached body and status. A reused key with a different payload is rejected with `409 idempotency_conflict`.
- `create-invoice` is one endpoint that does both draft and finalised creation (default `finalise: true` ‚Äî matches the user flow of "create + download" in one step). On finalisation: snapshots `bank_details`, `gst_config`, and `franchises.signature_url` onto the invoice row, and persists `grand_total_words` (Indian numbering ‚Äî lakh / crore aware). Tax routing is bill-to driven: `customer.state_code === franchise.state_code` ‚Üí CGST + SGST, else IGST. The schema's existing SQL helpers (`generate_invoice_no`, `get_active_bank`, `get_active_gst`) are called as RPCs.
- Read EFs all go through `userClient` so RLS does the multi-tenant gating; we never reach for the service role for reads.
